## 一、项目概述

### 1.1 项目背景
- **公司性质**：政府 IT 服务商  
- **服务对象**：四川省、成都市及下辖 23 个区县、部分街道等政府单位  
- **主营业务**：信息化建设及运营服务  
- **部门规模**：23 人（内部人力固定）  
- **项目数量**：约 30 个项目（存量+新增）  

### 1.2 当前阶段目标
根据老板口头需求，快速生成一个看板页面，用于**验证产品经理对业务的理解是否正确**。看板需让老板能够：
1. 清楚掌握部门经营情况  
2. 便于进行部门管理  

### 1.3 核心痛点
> **“成本怎么花的？花去哪些类目了？是不是在限定的额度内？”**

老板当前最迫切的需求是**成本管控**：
- 清楚每一笔成本的去向（按类目拆解）
- 实时监控成本是否超出预算/限额
- 快速定位超支项目或成本项

### 1.4 信息优先级
| 优先级 | 模块 | 老板关注点 |
|--------|------|------------|
| **P0** | 成本总览与结构 | 四大类成本实际 vs 计划、超支预警 |
| **P1** | 项目成本明细 | 哪些项目超支、超在哪个类目、关联售/PM/区县 |
| **P2** | 收入与利润概览 | 整体经营健康度（计划 vs 实际） |
| **P3** | 辅助查询 | 按销售/项目经理/区县筛选，未来扩展贡献值 |

## 二、用户角色（当前版本仅面向老板）

### 2.1 老板（核心用户）
- **核心需求**：掌握部门经营情况、进行成本管控
- **关注指标**：成本总览、项目成本明细、收入利润概览

### 2.2 项目经理（未来扩展）
- **核心需求**：跟进项目进程和成本花费，保证利润率
- **关注指标**：单个项目成本、利润率、进度、成本预警

### 2.3 销售（未来扩展）
- **核心需求**：签单目标完成情况、负责项目的利润率
- **关注指标**：目标完成率、项目利润率、项目数量

*注：V1 版本暂不实现项目经理、销售专属视图，但预留角色切换入口。*

## 三、成本管控逻辑详解

### 3.1 成本四大类目
| 类目 | 说明 | 管理属性 |
|------|------|----------|
| **内部人力** | 23 名固定员工的工资（老板、产品、运营、售前、销售、项目经理等） | 固定成本，整体展示简化；项目层面需记录人员投入，为未来 ROI 分析打基础 |
| **外采人力** | 项目需要时额外申请的研发、测试等外部人员费用 | 可变成本，严格监控，通常设项目级限额（如 ≤ 项目金额的 10%） |
| **商务费用** | 销售请客吃饭、送礼等 | 可变成本，需监控 |
| **其他** | 杂费（打车、打印）、媒体宣传费、招投标专家费等 | 可变成本，暂不细分，统一归入 |

### 3.2 管控指标（额度）
#### 3.2.1 整体层面
- 部门年度利润率 ≥ 20%（示例，可配置）

#### 3.2.2 项目层面
- 单个项目利润率 ≥ 30%（示例，可配置）
- 单个项目外采成本 ≤ 项目金额的 10%（示例，可配置）
- 单个项目商务费用 + 其他费用 ≤ 项目金额的 5%（示例，可配置）

#### 3.2.3 部门类目总额度
- 外采人力年度总额度 ≤ X 万元
- 商务费用年度总额度 ≤ Y 万元
- 其他费用年度总额度 ≤ Z 万元
- 内部人力为固定成本，无总额度限制（仅作信息展示）

#### 3.2.4 非项目成本
- 部门整体商务费用、其他费用需控制在一定额度内（与类目总额度一致，单独监控）

### 3.3 预警机制
- 当实际成本超过计划额度或达到阈值时，对应位置显示红色预警标识
- 预警分为：
  - **整体预警**：部门总成本超支
  - **类目预警**：某成本类目（外采/商务/其他）超过部门年度总额度
  - **明细预警**：项目/类目超支（如外采占比超限、项目利润率不达标）
- 超支项目自动高亮，支持快速下钻

## 四、数据模型（基于实际业务场景）

### 4.1 项目基础信息（来源于 CSV + 后续新增）
```typescript
interface Project {
  id: string                     // 序号或唯一ID
  name: string                   // 项目名称
  district: string               // 区域
  status: '进行中' | '已完成' | '待评估'   // 映射自 CSV 项目状态
  sales: string                  // 销售经理
  pm: string                     // 项目经理
  contractAmount: number         // 合同金额（签约金额）
  receivedPayment: number        // 已回款金额（从 CSV 已回款字段获得）
  progress: number               // 项目进度预估（0-100）
  projectType: '存量' | '新增'   // 项目类型：存量项目成本视为沉没成本，不计入系统明细
  actualCost?: number            // 存量项目的已发生总成本（仅作为历史参考，不参与明细分解）
  // 以下为新增项目特有字段
  costBreakdown?: {
    internalHR: number           // 分摊的内部人力成本（实时计算）
    externalHR: number           // 外采人力
    businessCost: number         // 商务费用
    otherCost: number            // 其他
  }
  // 部门统一标准
  planProfitRate: number         // 计划利润率（如 30%）
  externalHRRatioLimit: number   // 外采占比上限（如 0.1）
  otherCostLimit: number         // 其他可变成本上限（如 0.05）
}
```

**项目类型处理**：
- **存量项目**：签约日期在当年之前，其“已发生成本”已在 CSV 的“项目已发生成本”字段体现。系统仅展示该总成本，不分解明细，不作为成本管控对象（但会影响整体利润）。
- **新增项目**：当年新签约项目，所有成本均需从系统记录明细，参与预警与分摊。

### 4.2 成本项定义（支持二级）
```typescript
interface CostCategory {
  id: string
  name: string                   // 如“外采人力”
  parentId: string | null        // 支持二级，若为 null 表示一级分类
  isVariable: boolean            // 是否可变成本（内部人力为 false）
}
```
- **预设一级分类**：内部人力、外采人力、商务费用、其他  
- **二级分类示例**：外采人力下可细分“A公司人力”“B公司人力”，商务费用下可细分“餐饮”“礼品”等。  
- **V1 实现**：一级分类固定，二级分类暂不支持动态添加，但可在表格中通过“供应商”字段体现。

### 4.3 成本发生记录（从系统录入，V1 用 Mock 数据）
```typescript
interface CostRecord {
  id: string
  costCategoryId: string         // 关联到二级分类或一级分类
  projectId: string | null       // 若为 null 表示部门级成本
  amount: number
  occurDate: string              // YYYY-MM-DD
  supplier?: string              // 供应商/公司名称，用于二级明细
  remark?: string
}
```
**V1 数据来源**：
- 对于新增项目，成本记录通过系统录入（Mock 数据模拟）。
- 存量项目的“已发生成本”不进入成本记录表，仅在项目表中记录 `actualCost` 总金额。

### 4.4 内部人力成本分摊（数据来源于 OA 工时系统，V1 用 Mock 模拟）
```typescript
interface TimeEntry {
  id: string
  employee: string
  projectId: string | null        // null 表示部门级（如开会、团建）
  hours: number
  periodStart: string             // YYYY-MM-DD
  periodEnd: string
}
```
**分摊计算逻辑**：
- 已知部门年度内部人力总成本（固定值，从配置获取）。
- 按月或按统计周期，根据所有员工的工时比例，将内部人力总成本分摊到各项目和部门级成本。
- 分摊结果存入一个虚拟的成本记录（内部人力）中，作为项目的内部人力成本。
- **V1 实现**：由于工时系统尚未对接，使用 Mock 数据模拟工时记录，演示分摊逻辑。

### 4.5 部门配置（老板管理，V1 写死，V2 提供配置界面）
```typescript
interface DepartmentConfig {
  year: number
  totalPlanRevenue: number        // 部门年度计划总收入
  planProfitRate: number          // 部门年度计划利润率
  internalHRTotalCost: number     // 年度内部人力总成本（固定）
  costCategoryLimits: Map<string, number>  // 各一级成本项的年度总额度
  projectDefaults: {
    externalHRRatioLimit: number  // 外采占比上限（如 0.1）
    otherCostLimit: number        // 其他可变成本上限（如 0.05）
    minProfitRate: number         // 项目利润率下限（如 0.3）
  }
}
```

### 4.6 实时计算汇总
- **部门总成本** = 所有成本记录（包括内部人力分摊后）的总和 + 存量项目已发生成本
- **各项目成本** = 归属该项目的成本记录之和（含分摊的内部人力）
- **项目利润率** = (合同金额 - 项目总成本) / 合同金额
- **成本项完成率** = 实际发生额 / 年度总额度

## 五、功能模块详细设计

### 5.1 P0：成本总览与结构

#### 5.1.1 核心指标卡片
三个卡片横向排列：
- **总成本**：实际总成本 vs 计划总成本，完成率，进度条
- **可变成本合计**：外采人力 + 商务费用 + 其他，实际 vs 计划，完成率，进度条
- **固定成本（内部人力）**：仅展示实际金额，无计划对比（简化）

#### 5.1.2 成本结构图（环形图）
- 展示四大类成本的实际金额及占比（内部人力、外采人力、商务费用、其他）
- 支持点击图例显示/隐藏对应类目

#### 5.1.3 成本分类明细表（含部门总额度）
| 成本类型 | 部门总额度（计划） | 实际金额 | 差异 | 完成率 | 进度条 |
|----------|-------------------|----------|------|--------|--------|
| 内部人力 | - | ✓ | - | - | - |
| 外采人力 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 商务费用 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 其他 | ✓ | ✓ | ✓ | ✓ | ✓ |

- 内部人力行不显示计划额度，仅展示实际金额
- 差异正数（超支）红色，负数（节约）绿色
- 进度条根据完成率着色，超支部分红色

#### 5.1.4 整体利润预警
- 若当前部门实际利润率低于计划利润率，顶部全局提示条预警

### 5.2 P1：项目成本明细

#### 5.2.1 筛选区
- 状态筛选：Segmented（全部 / 进行中 / 已完成 / 待评估）
- 项目类型切换：存量 / 新增（帮助老板区分两类项目）
- 搜索框：项目名称 / 区县
- 多维度筛选：销售 / 项目经理 / 区县

#### 5.2.2 项目列表表格
| 字段 | 说明 | 样式增强 |
|------|------|----------|
| 项目名称 | 含区县信息 | 主文字加粗，区县信息 12px 灰色 |
| 类型 | 存量/新增 | 标签区分 |
| 状态 | Tag | 按状态配色 |
| 签约金额 | contractAmount | 右对齐，可排序 |
| 已回款 | receivedPayment | 右对齐，可排序 |
| 总成本 | 实际总成本 | 右对齐，可排序 |
| 外采成本占比 | 外采/签约金额 | 若超阈值显示红色背景 + ⚠️ |
| 利润率 | 实际利润率 | 若低于阈值显示红色背景 + ↓ |
| 进度 | progress | 进度条（高6px，圆角）+ 百分比 |
| 销售/项目经理 | 双行显示 | 销售（14px）、项目经理（12px 灰色） |
| 操作 | 详情 | 点击弹出抽屉（Drawer） |

- **存量项目特殊处理**：总成本直接显示 CSV 中的“项目已发生成本”，不展示成本占比和预警，利润率计算仍用合同金额与总成本计算。

#### 5.2.3 项目详情抽屉
- 成本明细：按四大类展示（内部人力、外采人力、商务费用、其他），若为存量项目则仅显示总成本。
- 二级成本明细：对于外采、商务等，展开显示供应商/明细列表（来自成本记录）。
- 人员投入（预留）：展示工时记录（Mock）。

## 七、技术约束与 AI 编码指引

### 7.1 Semi Design 使用规范
- 按需引入：配置 `babel-plugin-import`
- 主题定制：通过 `ConfigProvider` 注入 `designTokens`（主色 #4080FF，背景 #F8FAFC，错误 #EE6666 等）
- 组件约束：Table 固定表头+横向滚动；Drawer 宽度 480px；Segmented 用于状态筛选；Progress 圆角进度条

### 7.2 代码生成要求
1. 所有组件必须使用 Semi Design 组件库
2. 样式修改优先使用 `designTokens`，其次 CSS Modules
3. 图表使用 `echarts-for-react`，配置入场动画
4. 页面切换使用 React Router 6 + CSS 淡入淡出
